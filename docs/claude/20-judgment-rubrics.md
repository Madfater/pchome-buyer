# 判斷力 Rubrics

> 把「需要判斷」的時刻變成可核對的清單。每條附正例（該這樣）與反例（常見的錯）。例子多取自本專案，但規則通用。

## R1 何時升級模型

任一成立 → 升級（路徑見 [10-model-dispatch.md](10-model-dispatch.md) §升降級）：
- 錯誤次數達門檻（haiku 一次、sonnet 同一子任務兩次）。
- 問題涉及併發、時序、分散式狀態（本專案：run-group 執行緒、checkout lock、membership freeze、MAC 時效）。
- 兩個以上合理解釋且選錯代價高。
- 你發現自己在「試試看這樣行不行」而不是「我知道為什麼要這樣」。

**正例**：sonnet agent 兩次嘗試修 SSE 掉事件都失敗且錯誤不同 → 把兩次 diff＋錯誤輸出整理成失敗軌跡，派 `deep-reviewer` 分析根因，再決定修法。
**反例**：sonnet 第一次失敗，原因是打錯檔名這種低級錯 → 這不是能力問題，修正後同模型重試即可，升級是浪費。反過來也錯：同一個 agent 在同一個 bug 上重試第四次「說不定這次可以」——超過兩輪還在原地就是能力或方向問題，繼續重試只是燒 token。

## R2 何時算真的完成

全部成立才能回報完成，缺一條就明說缺哪條：
1. 對應的自動 gate 跑過且乾淨。**gate 清單的唯一正本是 CLAUDE.md「指令」區塊的「檢查」段**——照那份的註解判斷本次改動要跑哪幾項，不要憑記憶列舉（本檔刻意不重複清單，避免兩處漂移——踩坑紀錄見 00-diagnosis.md 教訓）。
2. **行為被實際執行過**：實跑受影響流程、或跑到測試、或派 `verifier` 實測。型別檢查過≠行為對。（本專案紅線：驗證不可觸及真實結帳，見 CLAUDE.md 不變量 7。）
3. 驗收由 fresh-context `verifier` 做過，逐條 PASS（無執行面的微小改動可走「輕量豁免」自驗，條件見 [10-model-dispatch.md](10-model-dispatch.md)）。
4. 沒有殘留的 TODO/暫時 hack/debug print；diff 裡沒有與任務無關的改動。
5. 回報裡附證據（跑了什麼、輸出關鍵行、檔案:行號），不是形容詞。

**正例**：「新增了 `/api/jobs/pause`：pyright 乾淨；用 `curl -X POST` 實測回 200 且 state 快照含新欄位；verifier 逐條 PASS（3/3）。」
**反例**：「已完成 pause 功能，程式碼看起來沒問題。」——沒有任何一條 gate 的證據；「看起來」不是驗證。另一個反例：跑了 build 但失敗，於是只回報「主要功能完成」——gate 紅著就不是完成，要如實回報失敗輸出。

## R3 何時停下來問使用者

任一成立 → 停下、整理現況、問（其餘一律自己決定並在回報中標明假設）：
- **不可逆或對外**：真實下單/付款、對外發送、刪除非自己建立的資料、force push。
- **花錢或動憑證**：任何會觸發 PChome 真實交易的操作；改動 `.env`、`auth_state.json`。
- **需求真歧義**：兩種解讀都合理、實作成本都高、選錯要重做（例：「把 job 卡合併」是指 UI 合併顯示還是 run-group 邏輯合併？）。
- **opus 也解不了**、或 deep-reviewer 與主結論衝突且無法用證據裁決。
- 發現使用者的前提可能錯了（要求修 A，但證據指向 A 沒壞、壞的是 B）→ 先回報發現，不要默默照做也不要默默改成修 B。

**正例**：任務是「測試結帳流程」→ 停下來問：用什麼商品測、可否接受進入 payinfo 頁但不點確認付款。因為這一步花真錢。
**反例**：為了「該用哪個變數名」「log 訊息要不要加時間戳」這種兩個答案都對、選錯零代價的事去問——這是把決策成本轉嫁給使用者，自己選一個、回報時提一句即可。

## R4 方向錯了的訊號（該換路，不是再試一次）

出現兩個以上 → 停止當前路線，寫下「目前假設是什麼、哪個證據推翻了它」，換一條路或升級：
- 同一個錯誤訊息在兩輪不同修法後原封不動。
- workaround 疊 workaround：每修一處就冒出新的一處要改。
- 要動的檔案數比最初估計膨脹超過一倍。
- 開始怪環境/工具/框架有 bug，卻沒有最小重現。
- 修法需要違反 CLAUDE.md 的不變量（例：想把 cart-modify 改成平行來提速——不變量 3 直接禁止，這條路本身就是錯的）。
- 解釋不了為什麼上一次會失敗，只是「再試一次看看」。

**正例**：修 cart 掉商品的 bug，改了兩處 retry 邏輯都沒用 → 回頭質疑假設，發現不是 retry 問題而是 RS store code 錯了（不變量 2 的 silent fail）→ 換路成功。
**反例**：pyright 報 10 個錯，逐個加 `# type: ignore` 壓掉——錯誤數在降所以感覺有進展，但這是 workaround 疊 workaround，真因（型別設計錯）沒動。

## R5 品質底線怎麼驗

- **檔案/文件**：派 `verifier` read-back——存在、涵蓋驗收條件、內鏈有效。自己重讀自己寫的東西不算驗證。
- **程式碼**：gate（R2.1）＋行為實跑（R2.2）＋diff 自查一遍「有沒有與任務無關的改動」。
- **高風險結論**（根因判定、架構選擇）：`deep-reviewer` 第二意見；或同題派兩個 agent 獨立解、比對。分歧無法用證據裁決 → R3 問使用者。
- **回報品質**：每個宣稱都要能回答「證據是什麼」。回答不了的宣稱降級為「推測」。

## R6 誠實極限（harness 補不了的事）

拆解、派工、fresh-context 驗證、多答案評審——這些補得了**執行品質**，補不了**品味與模糊題**：UI 文案語感、UX 取捨、「這樣設計優不優雅」、使用者沒說出口的偏好。

遇到時按序處理，不要硬做：
1. 能轉成可驗判準就轉（「好看」→「與現有 styles.css 的間距/色彩變數一致」）。
2. 轉不了 → 派 `deep-reviewer`（opus）出 2–3 個方案附取捨，讓使用者選；不要單方案硬上。
3. 連方案都列不出 → 明說「這題超出我能可靠判斷的範圍」，說明缺什麼。宣稱做不到不丟臉，假裝做得到才危險。

**正例**：「面板排版怪怪的，弄好看一點」→ 回：找出三個具體可改點（截圖佐證），列方案讓使用者挑方向。
**反例**：同樣的要求，直接大改 styles.css 兩百行然後宣稱「已優化視覺設計」——品味題上單方案硬上，大概率白做還汙染 diff。
